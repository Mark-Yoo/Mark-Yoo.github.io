OS

  • Job scheduling -> scheduler
    • context switching
  • process status(프로세스 상태)
    • process -> thread –> multithreading

Concurrency programming (동시성 프로그래밍)

  • Multithreading
    • CPU-bound
    • race-condition
    • dead-lock
  • Asynchronous I/O
    • I/O bound

Program vs Process

  • Program(image) - 하드 디스크에 저장되어있는 ‘하나’의 이미지
  • Process - 메인 메모리에 올라와서 실행을 시작한 프로그램

CPU scheduling

  • 선점형 스케쥴링(Premtive scheduling) : 더 높은 우선순위의 프로세스가 발생하면 CPU의 점유율을 회수해서 우선순위가 높은 프로세스에 강제 할당 하는 방식
    • Priority algorithm : 우선 순위가 높은 순서대로 실행
      • 우선순위가 다를 경우 채택
    • Round-robin algorithm : 우선순위를 따지지 않고 시간단위로 CPU를 할당해서 실행
      • 프로세스가 선점 다중 작업 시스템에서 실행할 수 있는 시간대
        • time slice
        • quantum
      • 우선 순위가 같을 때 채택
      • Priority가 너무 낮아서 실행 순서가 계속 밀릴 경우 starvation(기아 현상)이 생긴다. 이러한 starvation을 해결하기 위해 한 번씩 할당 된 프로세스가 끝날 때마다 낮은 우선순위의 프로세스에 aging을 통해서 priority를 점차 높여준다.
  • 비선점형 스케쥴링(Non-premtive scheduling) : 프로세스가 자원을 할당 받았을 경우, 자원을 스스로 반납할 때까지 계속 그 자원을 사용하는 방식

Context Switching - CPU의 레지스터에 저장되어 있는 값들을 바꾸는 방법

  • eax와 ecx에 들어간 값이 ALU를 거쳐 연산을 마치고 다시 eax로 돌아오는 과정을 거친다.
    1. mov eax, [a]
    2. mov ebx, [b]
    3. add eax, ecx <– eax = eax + ecx
    4. mov [c], eax
  • 연산 과정 중 다른 연산이 실행될 경우 이전에 실행되던 연산의 값은 PCB(Program Control Block)에 저장된다. 이 값은 이 연산이 다시 시작 될 경우 CPU에 그 값들을 다시 불러올 때 사용된다.

Context switching의 특징

  • 많이 일어날수록 시스템에 과부하가 일어난다. (느리고 무거운 작업)

  • 느리고 무겁지만 없앨 수 없는 과정. 시간을 너무 느리게 잡으면 멀티태스킹이 안됨, 반대로 너무 빠르게 잡으면 시스템이 과부하를 일으킴

Process vs Thread

  • Process
    • 실행 흐름
      • 실행흐름 -> 인스트럭션 실행 -> 함수 실행 -> stack frame
    • 각 프로세스에 4G의 ram 공간을 할당 : 실제 공간이 아닌 VAS(virtual address space), 가짜 메모리를 할당
  • Thread
    • 인스트럭션의 나열(기계어)

Concurrency programming (동시성 프로그래밍)

  • multi-thread 같지만 실제로는 CPU가 하나이기 때문에 thread 하나가 도는 동안 다른 스레드에 휴지 상태가 된다.

Parallel programming (병렬 프로그래밍)

  • single core multithreading : 코어가 하나라면 multi-thread와 single-thread와 속도가 비슷함
  • multi core multithreading : 여러개의 코어가 각각 thread를 담당하므로 속도가 빨라짐 (race condition의 위험이 있음)
    • race condition : 둘 이상의 입력 또는 조작의 타이밍이나 순서 등이 결과값에 영향을 줄 수 있는 상태. 입력 변화의 상태나 타이밍이나 순서가 예상과 다르게 작동하면 정상적인 결과가 나오지 않게 될 위험이 따름 -> 경쟁 위험

multi-threading vs single-thread

  • multi-threading은 속도는 빠르나 복잡 : 디버깅이 어려움
  • single-threading은 속도는 빠르지만 단순 : 디버깅이 쉬움
