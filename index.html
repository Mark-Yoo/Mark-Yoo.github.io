프로그래밍이란?

  • 해야할 일들의 순서
  • 목적은 문제의 해결
  • 프로그래밍 언어마다 순서가 달라짐
  • html은 tag와 attribute가 만들어져있어서 선언형 프로그래밍 언어로 구분하는 경우가 있다. (이미 정해진 tag와 attribute 이외의 것은 받아들이지 않는다.)
  • JS는 프로그래밍 언어 : 프로그래밍 언어는 개발자가 컴퓨터에게 하는 명령을 문서로 작성한 것. 컴퓨터와 커뮤니케이션을 하기 위한 언어. 자연어는 아닌 인공어.

  • 컴퓨터가 내 말을 어떻게 인식할 것인지에 대해서 항상 생각할 것.

  • 최근에는 사람이 잘 읽을 수 있는 코드를 만드는 사람이 코드를 잘 만드는 사람이 코딩을 잘하는 것.

  • 변수와 반복문은 악 = for문을 최대한 지양해야한다.

  • javasciprt: 인터프리터 언어(인간의 말을 컴퓨턱 알아들을 수 있게 변환을 도와줌) - 태생적으로 javascript는 느렸다. 하지만 google의 v8 엔진 이후로 비약적으로 빨라졌다.

  • 대부분의 프로그래밍 언어는 변수와 값, 키워드, 연산자, 표현식, 조건문과 반복문에 의한 흐름제어, 문, 문의 집합인 함수 그리고 객체, 배열 등의 자료구조와 같은 문법을 제공 (기본 개념이 같으므로 한 언어를 잘 배워놓으면 다른 언어의 학습에도 쓸 수 있다.)

자바스크립트의 탄생

  • 브랜던 아이크가 개발한 언어
  • Mocha -> Livescript -> Javascript
  • 마이크로소프트가 등장하면서 익스플로러를 보급하고 그러면서 넷스케이프에서 사용하는 JavaScript가 아닌 Jscript라는 비슷하지만 다른 언어를 만들어냄.
  • 결국 크로스 브라우징 이슈를 야기시킴.

자바스크립트의 파편화와 표준화

  • 향후에 생길 수 있는 크로스 브라우징 이슈를 해결하기 위해서 넷스케이프 쪽에서 마이크로소프트에 제안. ECMAScript(표준안)의 탄생. Javascript와 완전히 동일하지는 않음.
  • ES5는 flash의 사장과 함께 떠올랐다. 영상 삽입 등의 기능이 불가능했떤 HTML4가 이를 계기로 HTML5에 영상 삽입 등의 기능을 지원허기 시작함.
  • Javascirpt의 위상이 높아지면서 범용소프트웨어를 제작할 수 있는 범용언어가 되어가고 있다.
  • Ajax(Asynchronous JavaScript and XML) : 마이크로소프트에서 탄생. 구글에서 이를 이용해서 google maps를 만듦. 웹 어플리케이션의 가능성을 찾음.

JavaScript 성장의 역사

  • jQuery 크로스 브라우징을 기본적으로 지원한다. Javascript의 어려운 문법을 간단하게 바꿔준다.
  • 하지만 큰 문제점이 존재 : 향후 추가 예정
  • 브라우저에서만 동작하던 V8 엔진을 빼서 그대로 만든 것이 node.js -> javascript로 브라우저 환경 뿐만이 아닌 오프라인 환경에서도 사용가능한 앱을 제작할 수 있는 세상이 열림.
  • 다양한 일들을 할 수 있지만 인터프리터 언어라는 특성상 다른 언어들에 비해서 약간 느림.

Javascript와 ECMAScript의 차이

  • 모든 언어들은 API를 지원함 -> 이를 사용해서 코딩을 하게 되어있음.
  • ECMAScript와 클라이언트 사이드 Web API(브라우저 제조사에서 만듦)를 전부 포함해서 Javascript라고 부른다.
  • Javascript를 실행할 수 있는 환경은 두 가지 -> 브라우저(개발자 모드의 콘솔 창, html에 script를 추가하고 라이브 서버), Node.js
    • 브라우저 : ES + Web API (가장 기본적인 역할은 html과 css를 해석해서 렌더링 하는 것. Web API는 클라이언트 환경에서만 구동한다. 보안을 이유로 컴퓨터 안에 접근하는 것을 불허함)
    • Node.js : V8 엔진을 기반으로 ES + Node.js API (클라이언트에게서 온 레퀘스트에 리스폰스 하는 것이 주요한 기능이므로 Web API는 필요하지 않음. 브라우저 환경에 기반해서 만든 코드가 작동하지 않는 이유. 내 서버에서 작동하므로 파일에 접근하는 권한들이 부여됨.)
    • 양쪽에서 중요하게 작동하는 ECMAScript를 Core라고 부름. 제일 먼저 익히고 배워야 함. 그 이후에 Web API를 배워야 함. V8엔진은 ES만을 가지고 있고 Web API는 각 브라우저에서 가지고 있음.

자바스크립트의 특징

  • 인터프리터 언어. 컴파일은 하지만 실행파일을 만들지 않으므로 컴파일 언어라는 말은 맞지 않는다.
  • 명령형(imperative), 함수형(functional), 프로토타입 기반(prototype-based), 객체지향 프로그래밍을 지원하는 멀티 패러다임형 프로그래밍 언어이다.
    • 객체지향 프로그래밍이란 클래스로 쪼개어놓고 객체라는 것들이 서로 상호작용을 하면서 실행을 함 (매우 복잡한 방식) -> 함수형으로 회귀하자는 움직임이 있음
    • 대부분의 언어는 이 중 하나만을 제공하지만 javascript는 모두를 제공한다. 결국 효율성과 장단점을 구분하고 사용해야 하는건 프로그래머의 몫이 된다.
  • 패러다임에 대한 심도깊은 이해가 필요함 : 결국 모두 알아야 함

ES6 브라우저 지원 현황

NAT(Network address translation) vs NAPT(Network address port translation)

NAT

  • 라우터가 가진 공인 IP 주소를 하나씩 할당해줌 (소수의 IP주소를 가지고 있으므로 그 수가 모자람)

NAPT

  • 라우터가 가진 하나의 공인 IP 주소를 각각 다른 포트로 테이블을 만들어 호스트에게 할당. (같은 IP라도 포트의 번호가 다름)

Server

  • 특정 컴퓨터의 특정 프로세스 : socket이 많음
    • port: socket에 부여된 숫자
      • well-known port
    • socket
      • listening socket - 접속 요청할 때에 사용
      • data socket : 접속 후에 사용

TCP / UDP

  • TCP :
    • 신되로가 높음
    • 패킷 유실시 -> 재전송
    • 부담이 큼 (패킷을 보낸후 클라이언트가 받았는지 확인을 받아야 함)
  • UDP :
    • 신뢰도가 낮음
    • 패킷 유실시 -> 재전송 하지 않음
    • 부담이 적고 빠름 (동영상, 게임, 유튜브 등에서 사용)
    • 데이터 일부가 손실될 가능성이 있음

Application

  • HTTP(Hyper Text Transfer Protocol)
  • HTTPS- HTTP보다 보안이 더 강함
  • FTP
  • SMTP
  • DNS(Domain Name System) - Server IP를 사람이 읽기 쉽게 바꿔주는 프로토콜. 역으로 client가 도메인 이름으로 찾으면 이를 IP 주소로 바꿔 서버로부터 정보를 받아옴.

컨텐트 협상

  • 컨텐트 협상: client가 요청을 보내면 다양한 버전을 가지고 있는 server가 어떤 버전을 보낼것인지 결정하는 일. client가 요청한 자료가 없을 경우 대체제로 어떤 것을 보낼것인지에 대한 협상
    • 서버 주도 협상(server-driven) : 서버가 마음대로 정해서 대체제를 클라이언트에게 전달함
    • 에이전트 주도 협상(agent-driven) :
      클라이언트가 요구한 자료가 서버에 없을 경우 클라이언트에게 다시 리퀘스트를 요구함. 이를 통해서 클라이언트가 요구하는 대체제를 보냄. (두 번의 요청과 응답이 필요 -> 효율성이 올라감)

OS

    • process -> thread –> multithreading

Concurrency programming (동시성 프로그래밍)

  • Multithreading
    • CPU-bound
    • race-condition
    • dead-lock
  • Asynchronous I/O
    • I/O bound

Program vs Process

  • Program(image) - 하드 디스크에 저장되어있는 ‘하나’의 이미지
  • Process - 메인 메모리에 올라와서 실행을 시작한 프로그램

CPU scheduling

  • 선점형 스케쥴링(Premtive scheduling) : 더 높은 우선순위의 프로세스가 발생하면 CPU의 점유율을 회수해서 우선순위가 높은 프로세스에 강제 할당 하는 방식
    • Priority algorithm : 우선 순위가 높은 순서대로 실행
      • 우선순위가 다를 경우 채택
    • Round-robin algorithm : 우선순위를 따지지 않고 시간단위로 CPU를 할당해서 실행
      • 프로세스가 선점 다중 작업 시스템에서 실행할 수 있는 시간대
        • time slice
        • quantum
      • 우선 순위가 같을 때 채택
      • Priority가 너무 낮아서 실행 순서가 계속 밀릴 경우 starvation(기아 현상)이 생긴다. 이러한 starvation을 해결하기 위해 한 번씩 할당 된 프로세스가 끝날 때마다 낮은 우선순위의 프로세스에 aging을 통해서 priority를 점차 높여준다.
  • 비선점형 스케쥴링(Non-premtive scheduling) : 프로세스가 자원을 할당 받았을 경우, 자원을 스스로 반납할 때까지 계속 그 자원을 사용하는 방식

Context Switching - CPU의 레지스터에 저장되어 있는 값들을 바꾸는 방법

  • eax와 ecx에 들어간 값이 ALU를 거쳐 연산을 마치고 다시 eax로 돌아오는 과정을 거친다.
    1. mov eax, [a]
    2. mov ebx, [b]
    3. add eax, ecx <– eax = eax + ecx
    4. mov [c], eax
  • 연산 과정 중 다른 연산이 실행될 경우 이전에 실행되던 연산의 값은 PCB(Program Control Block)에 저장된다. 이 값은 이 연산이 다시 시작 될 경우 CPU에 그 값들을 다시 불러올 때 사용된다.

Context switching의 특징

  • 많이 일어날수록 시스템에 과부하가 일어난다. (느리고 무거운 작업)

  • 느리고 무겁지만 없앨 수 없는 과정. 시간을 너무 느리게 잡으면 멀티태스킹이 안됨, 반대로 너무 빠르게 잡으면 시스템이 과부하를 일으킴

Process vs Thread

  • Process
    • 실행 흐름
      • 실행흐름 -> 인스트럭션 실행 -> 함수 실행 -> stack frame
    • 각 프로세스에 4G의 ram 공간을 할당 : 실제 공간이 아닌 VAS(virtual address space), 가짜 메모리를 할당
  • Thread
    • 인스트럭션의 나열(기계어)

Concurrency programming (동시성 프로그래밍)

  • multi-thread 같지만 실제로는 CPU가 하나이기 때문에 thread 하나가 도는 동안 다른 스레드에 휴지 상태가 된다.

Parallel programming (병렬 프로그래밍)

  • single core multithreading : 코어가 하나라면 multi-thread와 single-thread와 속도가 비슷함
  • multi core multithreading : 여러개의 코어가 각각 thread를 담당하므로 속도가 빨라짐 (race condition의 위험이 있음)
    • race condition : 둘 이상의 입력 또는 조작의 타이밍이나 순서 등이 결과값에 영향을 줄 수 있는 상태. 입력 변화의 상태나 타이밍이나 순서가 예상과 다르게 작동하면 정상적인 결과가 나오지 않게 될 위험이 따름 -> 경쟁 위험

multi-threading vs single-thread

  • multi-threading은 속도는 빠르나 복잡 : 디버깅이 어려움
  • single-threading은 속도는 빠르지만 단순 : 디버깅이 쉬움

Binary Search (이진 탐색) VS. Linear Search (선형 탐색)

Linear Search

  • Linear search(선형 탐색)은 li를 순차적을 0번부터 끝까지 탐색을 하면서 target 값을 찾아내는 방법이다.
  • Linear search의 단점은 모든 자료를 처음부터 둘러봐야되는 점 때문에 자료의 양이 많아질수록 탐색 시간이 기하급수적으로 늘어난다.

Linear Search Example

  • li 에 주어진 value의 순서
  • index 값의 범위를 li의 length로 설정하고 index값이 li의 index값이 target과 동일할 경우 index값을 출력하는 linear search 공식
  • 5번째에 위치한 10을 찾아냄

Binary Search

  • Binary search(이진 탐색)은 sort를 사용하여 li 내의 값들을 순서대로 정렬하고 이 중 mid 값을 찾아 target 값과 비교한다. 만약 target 값보다 mid가 작다면 start 지점을 mid보다 1만큼 높게, target 값보다 mid가 크다면 end를 mid보다 1만큼 낮게 주고 다시 값을 찾는 연산을 반복한다.
  • linear search에 비해서 연산 속도가 비약적으로 빠르다는 장점을 갖는다.

Binary Search Example

  • 사전에 주어진 li 값과 target
  • binary search 기본 코딩
    • sort를 사용하여 li를 순서대로 정렬
    • start 값을 li의 0번째부터 시작
    • end 값은 start가 0번쩨 값부터 시작하므로 li length -1
    • start가 end보다 작거나 같을 경우 계속 search를 반복하는 while문을 사용
    • mid 값은 start와 end를 더한 값의 절반 (이번의 경우 start = 0, end = 6에서 시작하게 되므로 처음 mid 값은 3)
    • target이 mid의 값과 같을 경우 mid를 그대로 출력하는 return문
    • target이 mid보다 작을 겅우, end를 mid보다 1 작게, target이 mid보다 클 경우, start 값을 mid보다 1 크게하여 다시 while문을 돌린다.
  • 결과적으로 li는 순차적으로 정렬되어있고, 이 중 5번째인 10의 위치를 표시한다.

Call by reference VS Call by value

  • immutable object
    • 함수 내부에서 값을 변경할 수 없음

  • mutable object

    • list, dictionary, set
    • 새로운 객체를 할당하는 게 아니라면
    • 함수 내부에서 값을 변경할 수 있음

  • 만약 immutable object인 tuple 값을 바꿔야 한다면 tuple 내가 아닌 바깥 쪽에서 값을 받아오고 이를 return 함수를 이용해서 바꾸면 된다.

Call by reference

  • call by reference 방식은 value의 주소를 복사해서 쓰는 방식으로 value의 값이 바뀌더라도 주소만 그대로라면 value 값의 변화를 받아올 수 있다.
  • 바깥으로의 접근과 수정이 가능

Call by value

  • value 값을 ‘복사’ 해온는 방법. 실제로 보이는 값은 같아보이지만 독자적으로 그 복사값을 가지게 되므로, 원본 value 값이 바뀌더라도 변화가 없다.
  • 바깥으로의 접근이 불가

이 둘을 인자 전달 방식에 따른 함수 구분이라고 한다.
